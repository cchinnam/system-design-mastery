4 Deep Dive 1: Rules Engine & Chain Builder

Configurable Approval Rules

The Rules Engine determines WHO needs to approve and in what ORDER. Rules are configurable per tenant and per approval type.

Example: Discount Approval Tiers

  DISCOUNT APPROVAL RULES (SaaS Sales)
  ═══════════════════════════════════════════

  Discount 0-10%   → Sales Rep auto-approved
  Discount 10-20%  → Sales Manager approval
  Discount 20-30%  → Sales Director + Finance Manager (parallel)
  Discount 30%+    → VP Sales + CFO (sequential)

  DB ACCESS APPROVAL RULES (IT)
  ═══════════════════════════════════════════

  Read-only access  → Team Lead approval
  Read-write access → Team Lead → DB Admin (sequential)
  Admin access      → Team Lead → DB Admin → CISO (sequential)
  Production access → Team Lead → DB Admin → CISO → CTO (sequential)

Rule Storage

  • PostgreSQL: Rules table with conditions (JSON), approval chain definition, SLA config. Versioned — changes create new version, old approvals use old rules.
  • Redis cache: Hot rules cached with TTL 5 minutes. Cache key: tenant_id + approval_type. Invalidated on rule update.
  • Chain types: Sequential (A then B then C), Parallel (A and B simultaneously), Conditional (if amount > X, add C).

Chain Builder Output

  Example Chain for "30% discount on $50K deal":

  Step 1: VP Sales (Sarah)      ──sequential──>
  Step 2: CFO (Mike)            ──sequential──>
  Step 3: Legal Review (auto)   ──parallel with──>
  Step 3: Revenue Ops (Janet)

  Chain stored as ordered list of approval steps.
  Each step: approver_id, type (sequential/parallel), SLA_hours, escalation_to

5 Deep Dive 2: Async Notification Delivery

Kafka Topic Per Channel

  • Topic: notifications.slack — Consumed by Slack worker pods. Uses Slack Bot API to send rich messages with Approve/Reject buttons.
  • Topic: notifications.email — Consumed by Email worker pods. Uses AWS SES. HTML templates with one-click action links (tokenized URLs).
  • Topic: notifications.teams — Consumed by Teams worker pods. Uses Microsoft Graph API + Adaptive Cards with action buttons.

Idempotent Delivery

  • Deduplication key: approval_id + step_id + channel + attempt. If already sent, skip. Prevents duplicate notifications on Kafka consumer retries.
  • Delivery tracking: Each notification moves through states: queued → sent → delivered → read → acted-upon.
  • Multi-channel fallback: If Slack delivery fails after 3 retries, fall back to Email. User preference: primary channel (Slack) + fallback (Email).

Rich Notifications with Action Buttons

Slack Notification Example

Approval Request #4821
Type: Discount Approval
Requester: AI Sales Agent (on behalf of John Smith)
Details: 25% discount on Acme Corp deal ($50,000 ARR)
Justification: Competitive pressure from Competitor X, multi-year commitment
SLA: 4 hours remaining

[Approve] [Reject] [Request More Info] [Delegate]

6 Deep Dive 3: Escalation Scheduler

SLA Stages

SLA % Action Example (4-hour SLA)
75% Reminder notification to current approver At 3 hours: "Reminder: Approval #4821 needs your action"
100% Escalate to next level (approver's manager) At 4 hours: Notify manager, mark as escalated
150% Auto-action (approve/reject per policy) At 6 hours: Auto-approve if policy allows, else escalate to VP

Scheduler Implementation

  • 5-minute sweep: Cron job every 5 minutes scans all pending approvals. Checks current time vs. SLA deadlines. Triggers appropriate actions.
  • Redis sorted set: Pending approvals stored in ZSET with SLA deadline as score. ZRANGEBYSCORE finds all overdue items in O(log N + M).
  • Distributed lock: Redis SETNX ensures only one scheduler instance processes at a time. Prevents duplicate escalations.

OOO (Out of Office) Handling

  • Calendar integration: Check Google Calendar / Outlook for OOO status before sending notifications.
  • Auto-delegate: If approver is OOO, auto-delegate to their configured backup or manager.
  • SLA adjustment: OOO time doesn't count against SLA if delegation is configured. Otherwise, immediate escalation.
  • Return notification: When approver returns from OOO, send summary of delegated decisions.

7 Scaling & ML

Scaling Strategies

  • Kafka partitioning: Partition by tenant_id for ordered processing per tenant. Scale consumers horizontally per channel.
  • Redis cluster: Shard state data by approval_id. Hot approvals (pending) in Redis, completed approvals archived to PostgreSQL.
  • Channel workers: Auto-scale based on queue depth. Slack worker scales independently from Email worker.
  • Database: PostgreSQL with partitioning by created_date. Archive approvals older than 90 days to cold storage.

ML Enhancements

  • Predicted approval time: ML model predicts when each approver typically responds. Use to set smarter SLA deadlines.
  • Channel preference: Learn which channel gets fastest response per user (some respond to Slack in 5 min, Email in 2 hours).
  • Auto-approval candidates: Identify routine approvals that are always approved — suggest auto-approval rules to admins.
  • Optimal notification timing: Learn each approver's active hours. Don't send at 2 AM — queue for their work hours start.

8 Cheat Sheet

Notification & Approval — Key Numbers

  • 100K approvals/day, 250K notifications/day
  • <1s notification delivery latency
  • Kafka topic per channel (Slack, Email, Teams)
  • Idempotent delivery with dedup key: approval_id + step_id + channel
  • Rules Engine: configurable per tenant, cached in Redis (TTL 5 min)
  • Chain types: sequential, parallel, conditional
  • SLA stages: remind 75%, escalate 100%, auto-action 150%
  • 5-min escalation sweep using Redis sorted set
  • State machine: submitted → pending → approved/rejected/escalated/expired
  • OOO handling with calendar integration and auto-delegation
